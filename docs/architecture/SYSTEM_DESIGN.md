# System Design (Current Baseline)

This document describes the implemented technical design as of 2026-05.

## Quick Path

1. Read `REQUIREMENTS.md` for product/compliance intent.
2. Use this file for implementation architecture and runtime flows.
3. Use ADRs (`docs/adr/INDEX.md`) for decision rationale and change history.

## Context

The platform provides electronic records processing with three modules:
- Citizen frontend (`front-end`) — Angular 19
- Backoffice frontend (`back-office`) — Angular 19
- Core backend (`backend`) — Spring Boot 3.x

The system is developed with H2 in dev/test and prepared for PostgreSQL in production.

## Runtime Architecture

| Layer | Module | Main Responsibilities |
|---|---|---|
| Gateway | Nginx (Reverse Proxy) | SSL termination, subdomain routing, static asset caching |
| Presentation | Citizen UI | Public catalog, authentication, procedure start, case wizard |
| Presentation | Backoffice UI | Dashboard, tasks, users, procedure management, i18n management |
| Application/Core | Backend API | JWT auth, RBAC, case/procedure logic, catalog i18n, BPM integration |
| Data | PostgreSQL/H2 | Domain persistence + `procedure_type_i18n` translations |
| Messaging | SMTP + Mailpit | Local transactional email delivery and inspection |
| Observability | Loki + Grafana + Prometheus | Log aggregation, dashboards, metrics collection |

## Key Flows

### 1) Citizen Starts a Procedure

1. Citizen selects a catalog item.
2. UI navigates to protected route `/expedientes/nuevo/:procedureId`.
3. If unauthenticated, guard redirects to `/sede/login?returnUrl=...`.
4. After login, user is returned to original wizard route.
5. Wizard resolves procedure by stable identifier and creates case using `procedureId`.

### 2) Catalog Localization

1. Frontends send `Accept-Language`.
2. Backend resolves locale (supported list + fallback).
3. Catalog localization service looks up DB translations in `procedure_type_i18n`.
4. If missing, backend falls back to message bundles.

### 3) Backoffice Protected Operations

1. UI calls protected `/admin/*` endpoints with access token.
2. If token is expired, backend returns `401`.
3. Backoffice interceptor refreshes token via `/auth/refresh` and retries original request.

### 4) Transactional Email (Local Runtime)

1. Backend emits verification/notification email through `JavaMailSender`.
2. SMTP delivery goes to Mailpit (`mailpit:1025` in Compose network).
3. Developers/QA validate delivery and content in Mailpit UI (`http://localhost:8025`).

## Data Design: Catalog i18n

Table: `procedure_type_i18n`

| Column | Type | Notes |
|---|---|---|
| `id` | UUID | PK |
| `procedure_type_id` | UUID | FK -> `procedure_types.id` |
| `locale` | VARCHAR(10) | Locale key (`es-ES`, `ca-ES`, ...) |
| `title` | VARCHAR(255) | Localized title |
| `description` | TEXT | Localized description |
| `unit` | VARCHAR(100) | Localized unit |
| `created_at` | TIMESTAMPTZ | Auditable metadata |
| `updated_at` | TIMESTAMPTZ | Auditable metadata |

Constraints:
- Unique `(procedure_type_id, locale)`
- Index by `locale`

## Security Design

- Stateless JWT auth.
- Coarse endpoint control in `SecurityConfig`.
- Role model: `ROLE_CITIZEN`, `ROLE_TRAMITADOR`, `ROLE_ADMIN`.
- Public read-only catalog endpoint is allowed without auth.
- Unauthenticated requests to protected resources return `401`.

## Deployment Notes

- Production profile uses PostgreSQL with `ddl-auto: validate`.
- Required SQL migration for i18n table:
  - `backend/db/postgresql/001_create_procedure_type_i18n.sql`
- Dev profile uses H2 with `create-drop` for fast iteration.

### 5) Case Submission (Entry Registry + Summary Document)

1. Citizen completes wizard flow and calls `POST /citizen/procedures/{id}/submit`.
2. Backend signs all pending citizen-uploaded documents via `SignatureService` (CMS/PKCS#7).
3. `RegistryService.generateEntryNumber()` atomically generates NRE (Numero de Registro de Entrada) in format `RE/{unitCode}/{year}/{seq6}` using `INSERT ... ON CONFLICT DO UPDATE` on `document_registry_counters`.
4. Procedure gets `SUBMITTED` status, `submittedAt`, `recordNumber` (`EXP/{unitCode}/{year}/{seq6}`), and `entryNumber` (`RE/...`).
5. `SummaryDocumentService` generates a PDF with:
   - Form data labels (resolved from `ProcedureTask.formSchema` JSON fields)
   - Attachment list with sizes
   - Registry entry block (NRE, date, unit)
   - Signed by server via `SignatureService.signDocument()`
   - Assigned NRS (Numero de Registro de Salida) via `RegistryService.generateExitNumber()` — format `RS/{unitCode}/{year}/{seq6}`
   - Registered in `PublicSignatureVerificationService` for CSV-based verification
6. BPMN workflow starts via Flowable with the submitted procedure.
7. ENI metadata is persisted for both procedure and all documents.

### 6) Document Numbering Scheme

| Type | Prefix | Applies To | Generated By |
|------|--------|-----------|--------------|
| Record Number | `EXP/` | Procedure (on submit) | `CaseServiceImpl.generateRecordNumber()` |
| Entry Number (NRE) | `RE/` | Procedure (on submit) | `RegistryService.generateEntryNumber()` |
| Exit Number (NRS) | `RS/` | System-generated documents | `RegistryService.generateExitNumber()` |
| Document Number | `DOC/` | User-uploaded documents | `DocumentServiceImpl.uploadDocument()` (TODO) |

## Data Design: Registry Counters

Table: `document_registry_counters`

| Column | Type | Notes |
|--------|------|-------|
| `registry_type` | VARCHAR(10) | `ENTRY` or `EXIT` |
| `unit_code` | VARCHAR(20) | Administrative unit code |
| `year` | INTEGER | Counter year |
| `last_value` | BIGINT | Last assigned sequence number |

Constraints:
- PK: `(registry_type, unit_code, year)`
- Atomic increment via `INSERT ... ON CONFLICT DO UPDATE ... RETURNING last_value`

## Data Design: Entry/Exit Numbers

- `procedures.entry_number` — NRE assigned on submission.
- `documents.exit_number` — NRS assigned to system-generated documents (summary, notifications).
- `documents.generated` — boolean flag identifying auto-generated documents vs. citizen uploads.
